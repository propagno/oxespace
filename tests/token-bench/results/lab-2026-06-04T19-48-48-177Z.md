# Semantic-Search Lab — one request, with vs without

**Request:** _Como o servidor MCP entra em hibernação por inatividade e onde fica o timer de sleep?_

- Model: **Xenova/all-MiniLM-L6-v2** (quantized) · same pipeline the app ships
- Corpus: **256** indexable files / **2369** chunks under `electron`, `src`, `shared` · Tokenizer: **gpt-tokenizer (o200k_base — approximates Anthropic)**
- Ranking: chunked (~800 chars/chunk), best-chunk cosine — same as the shipped service
- Top-K returned by semantic search: **5**

## Bottom line

- **Semantic search** is the dominant lever: it replaces a broad context with a small top-5 set — **453,846 → 19,915** vs a repo dump (**−95.6%**), **57,976 → 19,915** vs grep (**−65.6%**).
- **RTK** removes terminal spam: **−53.2%** on the command output it sees.
- **Caveman** trims the answer: **−82.0%** output tokens.
- **All three together (vs grep baseline):** **58,764 → 20,220** tokens — **−65.6%**.

## Ranking quality

Top hit 0.386, #5 0.263. Chunked best-chunk ranking lets a query match logic anywhere in a file (not just its header), which the previous whole-file embedding could not.

Remaining consideration: `all-MiniLM-L6-v2` is English-centric, so a Portuguese query against English code still retrieves a bit weaker than an English query would — swapping in a multilingual MiniLM would close that gap. The token-saving accounting above is independent of ranking order.

> Token counts are an offline estimate (the cross-strategy comparison is the signal). Semantic ranking is real; the verbose/terse answer pair is illustrative.
