# Semantic-Search Lab — one request, with vs without

**Request:** _Como o servidor MCP entra em hibernação por inatividade e onde fica o timer de sleep?_

- Model: **Xenova/multilingual-e5-small** (quantized) · same pipeline the app ships
- Corpus: **382** indexable files / **2632** chunks under `electron`, `src`, `shared` · Tokenizer: **gpt-tokenizer (o200k_base — approximates Anthropic)**
- Ranking: chunked (~1500 chars/chunk), best-chunk cosine — same as the shipped service
- Top-K returned by semantic search: **5**

## Bottom line

- **Semantic search** is the dominant lever: it replaces a broad context with a small top-5 set — **851,673 → 18,650** vs a repo dump (**−97.8%**), **103,258 → 18,650** vs grep (**−81.9%**).
- **RTK** removes terminal spam: **−53.2%** on the command output it sees.
- **Caveman** trims the answer: **−82.0%** output tokens.
- **All three together (vs grep baseline):** **104,046 → 18,955** tokens — **−81.8%**.

## Ranking quality

Top hit 0.889, #5 0.860. Chunked best-chunk ranking lets a query match logic anywhere in a file (not just its header), which the previous whole-file embedding could not.

Model is now multilingual (`Xenova/multilingual-e5-small`, 768-dim, 512-token window), which closes the Portuguese-query-vs-English-code gap the English-only MiniLM had. The token-saving accounting above is independent of ranking order.

> Token counts are an offline estimate (the cross-strategy comparison is the signal). Semantic ranking is real; the verbose/terse answer pair is illustrative.
